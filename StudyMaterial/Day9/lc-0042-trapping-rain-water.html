📋 Problem + Core Insight

🎯 LeetCode 42: Trapping Rain Water

The Core Insight: Water at each position = min(max height to the left, max height to the right) - height at position.

Instead of computing full prefix arrays, use Two pointers to track these maximums inward in O(1) space.

📋 Problem

Given n non-negative integers representing an elevation map

where the width of each bar is 1, compute how much water it can trap after raining.

height = [0,1,0,2,1,0,1,3,2,1,2,1]
→ 6

Constraints: 1 ≤ n ≤ 2 * 10⁴ | 0 ≤ height[i] ≤ 10⁵

📊 Complexity Analysis

🔬 Complexity Analysis: DP Array vs Two Pointers

The Core Insight: Precomputing left_max and right_max arrays takes O(n) space.

By using two pointers advancing inward, we track the running maximums on the fly, reducing memory overhead essentially to zero.

Approach Time Space Notes
Prefix/Suffix Max (DP) O(n) O(n) Requires 3 complete array scans
Two Pointers O(n) O(1) One scan inward, highly efficient

🚀 Optimal Solution

def trap(height: list[int]) -> int:
    left, right = 0, len(height) - 1
    left_max = right_max = 0
    water = 0

    while left < right:
        if height[left] <= height[right]:
            # Right side is at least as tall \u2014 left_max determines water level
            left_max = max(left_max, height[left])
            water += left_max - height[left]
            left += 1
        else:
            # Left side is taller \u2014 right_max determines water level
            right_max = max(right_max, height[right])
            water += right_max - height[right]
            right -= 1

    return water

🎤 Interview Q&A

🎤 5 Interview Q&A

Q1: Why does the two-pointer approach work?

Answer: When height[left] <= height[right], the right boundary is guaranteed ≥ left_max.

So water at left is exactly left_max - height[left]. We don't need to know the actual right max — just that it's ≥ left_max.

Q2: What's the O(n) space solution?

Answer: Precompute prefix_max (left max at each index) and suffix_max (right max) into full arrays,

then iterate once doing min(prefix_max, suffix_max) - height. Two pointers does the exact same logic in O(1) space.

Q3: How does this relate to data engineering?

Answer: "Windowed capacity headroom" — for each time period, what is the effective available capacity

bounded by upstream and downstream constraints? The algorithm isolates the true restricting bottleneck.

Q4: What happens if the array is continuously increasing or decreasing?

Answer: No water is trapped. The pointer on the shorter side will just continually move inward,

updating its max but never finding a valley. For example, in [1,2,3,4], left moves rightward, left_max updates, but difference is always 0.

Q5: Could you solve this with a Monotonic Stack instead?

Answer: Yes. A decreasing monotonic stack can catch valleys and compute horizontally-bounded water pools

when a taller wall is encountered. However, it takes O(n) space for the stack, making the Two Pointers approach strictly better for space.
