LC #0001 — Two Sum Easy
HashMap Pattern

📊 The Problem

Given an array of integers `nums` and an integer `target`, return indices of the two numbers such that they add up to `target`.

Example: nums = [2, 7, 11, 15], target = 9

Goal: Find exactly one pair of distinct indices (i, j) where nums[i] + nums[j] == target.

💡 Key Insight

We need to check every pair. A nested loop checks $O(n^2)$ pairs, but instead of scanning forward for the complement, we can instantly look backward in $O(1)$ time by storing visited numbers in a HashMap.

The Party Guest List:
Instead of asking every guest "Are you my date?", you write down the name of who you're looking for (your complement) and check if they've already signed the guestbook (the hash map). If they have, you've found your match instantly!

Why this works:

  • The equation is `A + B = Target`, which means `B = Target - A`.
  • We can calculate `B` (the complement) the moment we see `A`.
  • Hash tables allow $O(1)$ average time complexity for lookups.

🔧 How the Algorithm Works

1 Initialize a Hash Map

Create an empty dictionary `seen` to store `{value: index}` mapping of numbers we've iterated past.

2 Iterate the Array

Loop through `nums` tracking both the index `i` and the value `num` using `enumerate`.

3 Calculate the Complement

Compute `complement = target - num`. This is the exact value required to solve the problem.

4 Check the Hash Map

If the `complement` is already in the `seen` hash map, we are done! Return `[seen[complement], i]`.

If not, store the current number in the hash map (`seen[num] = i`) for future numbers to look back at, and continue iterating.

🎯 The Hash Map Trick

Store elements backward to achieve $O(1)$ lookup

Target: 9
num = 7 at index 1

Calculation:

• complement = 9 - 7 = 2
• `if 2 in seen`: YES, `seen[2]` maps to index `0`
• Return [0, 1]

This guarantees we only visit each element exactly once, shifting the bottleneck from comparison to mere storage allocation.

⚡ Why It's Optimal

❌ Brute Force
A nested loop compares every element against every other subsequent element.

Time: O(n²)
Space: O(1)
✅ HashMap (One-Pass)
Stores previously seen elements and their indices in a hash map mapping `val -> idx`.

Time: O(n)
Space: O(n)

The time complexity drops drastically because hash map lookups take $O(1)$ time on average. Trading $O(n)$ space for $O(n)$ time is the gold standard of data engineering!

📖 Algorithm Explanation in Plain English

As you walk down the list of numbers, you constantly ask: "What number do I need to reach the target?" You immediately look in your notebook (the hash map) to see if you passed that number earlier. If you did, you have found the pair! If not, you write down your current number and its position in the notebook, so that numbers further down the line might find you later.

Step 1: First Element

Process the first number.

Tracking [2, 7, 11, 15] with Target 9:
i=0, num=2. Complement is 7.
Is 7 in seen? No. Store `{2: 0}`.
Step 2: The Match

Process the next number and check for the complement.

Next Iteration:
i=1, num=7. Complement is 2.
Is 2 in seen? Yes! Return `[seen[2], 1]`, which is `[0, 1]`.
🔑 The Confusing Part: What if the array has duplicates?

If the input is `nums = [3, 3], target = 6`, won't the hash map overwrite the first `3`?

Scenario:

i=0, num=3. Calculate comp=3.
`seen` is empty. Add `{3: 0}`.

Why This Matters: The second `3` (at `i=1`) computes `comp=3`. It checks if `3` is in `seen` *before* attempting to add itself to the map! It finds the *first* `3` and correctly returns `[0, 1]` without overwriting.

💡 The Genius Move:
Checking the map *before* adding the current element prevents an element from matching with itself.

📝 Annotated Solution

def twoSum(nums, target): # seen maps a value to its index: {value: index} seen = {} for i, num in enumerate(nums): # Calculate the exact value we need to reach the target complement = target - num # Perfect O(1) average lookup if complement in seen: return [seen[complement], i] # Store the current number's index for future iterations seen[num] = i return [] # Default return if no solution is found (though the problem guarantees one)
Magic Line: if complement in seen:
This performs an O(1) hash map lookup instead of a slow nested O(N) linear search for the complement!

🔍 Example Walkthrough

Input: `nums = [3, 2, 4], target = 6`

Index 0, num = 3:
complement = 6 - 3 = 3.
3 not in `seen`. Store `seen[3] = 0`.
State: `seen = {3: 0}`
Index 1, num = 2:
complement = 6 - 2 = 4.
4 not in `seen`. Store `seen[2] = 1`.
State: `seen = {3: 0, 2: 1}`
Index 2, num = 4 — BLOCKER FOUND:
complement = 6 - 4 = 2.
2 *is* in `seen`! `seen[2]` maps to `1`.
Pair found.
Return result:
Return `[seen[2], 2]` -> `[1, 2]`.

Answer: [1, 2] — Using a one-pass approach successfully locates the target complement backwards in O(1) lookup.

🎤 Interview Q&A

Questions an interviewer will actually ask:

Q: What is the time and space complexity?
A: Both are O(N). Time is O(N) because we iterate the array exactly once, and Python hash table lookups are O(1) on average. Space is O(N) because the worst-case scenario stores N items in the `seen` hash table.
Q: What data structure is `{}` in Python under the hood?
A: A hash table (dictionary). It uses a hash function to map keys to an array index, drastically optimizing insertion and lookups to average-case O(1) time.
Q: What if there were multiple valid pairs? How do you return all of them?
A: We would initialize `results = []` at the beginning. Instead of `return`, we would `results.append([seen[complement], i])` when a pair is found. We'd return `results` at the end of the iteration.
Q: How would you solve this if the array was already sorted?
A: I would use a Two-Pointer approach. Place a `left` pointer at index 0 and a `right` pointer at the end of the array. If the sum is too small, I increment `left`. If too large, I decrement `right`. This runs in O(N) time with O(1) space!
